1Objetivo

django-agent es un paquete Django reutilizable que agrega un widget de chat flotante (abajo a la derecha) a cualquier app/admin de Seenka. Detrás hay un agente LLM (Gemini en Vertex) que:

  • Entiende en qué página está el usuario.
  • Mantiene la conversación persistente entre navegación y recargas.
  • Puede recibir contexto/conocimiento y, con el tiempo, aprender.
  • Puede ejecutar acciones sobre los datos de la app (con permisos + confirmación + auditoría).

Caso de uso disparador: en el admin de panelists, pegar credenciales en cualquier formato y que cree los panelistas/cuentas — sin scripts.

4Flujo de un mensaje

  1. El widget carga en la página → GET /agent/history trae la conversación activa del usuario.
  2. El usuario escribe → POST /agent/message con {text, page_context}.
  3. El backend arma el prompt: system (instrucciones + contexto de app + page-context + memoria) + historial + mensaje, y declara las tools disponibles para ese usuario.
  4. Gemini responde texto o un functionCall.
  5. Si es una tool de lectura → se ejecuta y el resultado vuelve al modelo para la respuesta final.
  6. Si es una tool de escritura (confirm=True) → no se ejecuta: el backend devuelve al widget una tarjeta de confirmación con el preview de lo que haría.
  7. El usuario confirma → POST /agent/confirm → se chequea permiso, se ejecuta, se escribe en AuditLog, el resultado vuelve al modelo → respuesta final.
  8. Cada turno (usuario, asistente, tool) se persiste en Message.
La confirmación es el corazón de la seguridad: ninguna escritura ocurre sin un “sí” explícito del usuario sobre un preview concreto.

5Gemini en Vertex AI

Autenticación

Vía Application Default Credentials: la service account del pod. Sin API key. Requiere que esa SA tenga el rol roles/aiplatform.user en infoxel-tagx. acción de infra

SDK y cliente

from google import genai
from google.genai import types

client = genai.Client(vertexai=True, project="infoxel-tagx", location="us-central1")

resp = client.models.generate_content(
    model="gemini-3.5-flash",
    contents=history,                       # turnos previos + mensaje
    config=types.GenerateContentConfig(
        system_instruction=system_prompt,   # rol + page-context + memoria
        tools=[tool_declarations],          # function declarations
        temperature=0.2,
    ),
)

Function-calling (loop del agente)

  1. El modelo devuelve functionCall(name, args).
  2. El backend resuelve la tool en el ToolRegistry, valida permiso y (si escribe) confirma.
  3. Se devuelve functionResponse al modelo y se repite hasta que responde texto.
Modelo: gemini-3.5-flash (rápido y barato, suficiente para extracción + tool-use). Configurable por setting.

6CRUD automático sobre modelos plug-and-play

No hay que declarar tools. El agente opera genéricamente sobre los modelos registrados en el admin de cada app. Instalar = agregar a INSTALLED_APPS + incluir las URLs. Cero código por modelo.

Descubrimiento

El ModelRegistry usa admin.site._registry: lo que está registrado en el admin es lo que el agente ve. Scoping natural — si un modelo no está en el admin, el agente no lo toca.

Operaciones genéricas (las “tools” que ve Gemini)

En vez de N tools por modelo, un puñado de funciones genéricas donde model es un parámetro — escala a cualquier cantidad de modelos sin inflar las declaraciones:

FunciónPermiso DjangoConfirma
describe_models() — modelos y sus campos—no
query(model, filtros, orden, límite)viewno
get(model, pk)viewno
create(model, data)addsí
update(model, pk, data)changesí
delete(model, pk)deletesí

Schemas de campos

Derivados de model._meta: nombre, tipo, requerido, choices, y FKs (el agente resuelve la FK por nombre natural o pk). Los JSONField (ej. el metadata de Account) se escriben como objeto.

Permisos automáticos

Cada operación chequea el permiso Django del modelo (view/add/change/delete) contra request.user. Si no tiene permiso, esa operación no se ofrece. Seguridad plug-and-play — reusa el sistema de permisos que ya existe.

El caso “crear cuentas en cualquier formato”

Sale gratis del CRUD genérico: el LLM extrae los campos del texto pegado (lo que sea) y encadena create(Panelist, …) + create(Account, {…, metadata:{…}}). El parsing flexible lo hace el modelo, no regex, y sin tool dedicada.

Escape hatch (opcional): si un modelo necesita lógica especial (idempotencia, derivaciones), una app puede registrar una tool custom. Es opcional, no requerido — el default es CRUD automático.

7Seguridad MVP

  • Permisos: cada tool declara un permiso Django; se evalúa contra request.user. Sin permiso, la tool ni se ofrece al modelo.
  • Confirmación: toda tool con confirm=True muestra un preview y exige “Confirmar” antes de ejecutar. El modelo nunca escribe solo.
  • Audit log: toda ejecución guarda usuario, tool, args, resultado, timestamp.
  • Scope: el agente solo ve las tools de las apps instaladas y permitidas para ese usuario.
  • Staff-only: el widget solo se monta para usuarios is_staff (configurable).
Regla dura: el LLM propone, el usuario confirma, el backend ejecuta con permisos. Ninguna acción destructiva sin confirmación.

9Widget & page-context

  • JS/CSS vanilla servidos por el paquete; se inyectan vía un fragmento en el base template del admin (o un context processor).
  • Burbuja abajo a la derecha; abre un panel con historial + input + tarjetas de confirmación.
  • Page-context que manda con cada mensaje: url, path, vista resuelta, y si está editando un objeto: model + pk. El backend lo puede enriquecer (traer el objeto).
page_context = {
  "url": "/admin/panelists/account/",
  "view": "panelists.account.changelist",
  "object": null            // o {"model":"panelists.account","pk":16}
}

10Estado & persistencia

El estado vive server-side (modelos Conversation/Message), no en el navegador. Por eso sobrevive a:

  • Cambio de página: el widget vuelve a montar y trae el historial por GET /agent/history.
  • Recarga / otra pestaña: misma conversación (clave: user).

El page-context cambia con cada página; la conversación es continua.

11Alcance del MVP

Incluye

  • Paquete instalable plug-and-play + widget flotante (staff-only, solo admin).
  • Backend Gemini 3.5 Flash (Vertex) con function-calling.
  • CRUD automático sobre los modelos del admin (ModelRegistry), con permisos Django.
  • Persistencia de conversación — una por usuario (sobrevive navegación/recarga).
  • Page-context.
  • Seguridad: permisos + confirmación + audit log.
  • Validado con el caso real: crear cuentas en panelists (vía CRUD genérico).

Fuera del MVP

  • Conocimiento/RAG, memoria/aprendizaje, storage compartido entre apps, multi-conversación, streaming, tools custom.

12Fases futuras

  1. Conocimiento — store de contexto por app + recuperación (RAG si crece).
  2. Aprendizaje — memoria de hechos + feedback 👍/👎 recuperado por turno.
  3. Storage compartido — externalizar memoria/conocimiento entre apps (solo storage).
  4. UX — streaming, multi-conversación, búsqueda en historial.
